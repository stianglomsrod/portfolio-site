# DNB Content Outline

## Purpose
Translate the Klar -> DNB skills distillation into a concrete content outline for the DNB AI-First Engineering portfolio variant.

This is a content-planning document only. No UI, routing, component, styling, or deployment implementation is included.

## Positioning guardrails
- Position Stian as an AI-first builder / software engineering candidate with strong system understanding.
- Emphasize practical fullstack product-building understanding: frontend, backend, APIs, database, auth, frameworks, CSS libraries, deployment, and how these layers connect.
- Emphasize AI-assisted prototyping and development, not only AI-assisted prototyping.
- Emphasize high sustained learning/work capacity through evidence: learning velocity, output volume, cross-domain synthesis, persistence, and conversion of new knowledge into working prototypes, documentation, workflows, and systems.
- Treat lower-level engineering as a growth direction, not established senior expertise.
- Keep tone concrete, human, technically curious, honest, and useful for DNB AI Tech.
- Exclude private/medical framing.

## 2. How I build with AI
### Section objective
Show a concrete, professional, safe agentic workflow that is useful to an engineering team, not a generic "AI enthusiasm" statement.

### Core narrative points
- Start with context packaging: product goal, constraints, evidence source, and acceptance criteria.
- Convert requirements into small, reviewable implementation chunks.
- Use agent output as draft implementation, not final truth.
- Run review and correction loops: check assumptions, compare against requirements, tighten scope.
- Document decisions and tradeoffs continuously so work remains inspectable and transferable.
- Apply QA and safety gates before claiming completion.
- Track risks explicitly: unsupported claims, missing tests, and unknowns.

### DNB AI Tech relevance
- Demonstrates safe AI-first engineering behavior that can scale across teams.
- Supports enablement: workflows others can reuse, not one-off hero execution.
- Balances speed with quality through explicit guardrails and verification.

### Concrete language style
- Use examples from actual workflow artifacts: onboarding docs, handoff protocol, QA checklist, implementation log.
- Avoid corporate abstraction and buzzwords.
